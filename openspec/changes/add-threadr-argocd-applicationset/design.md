## Context
Threadr already has production-oriented Kustomize overlays for the control plane and worker, and the main image workflow updates the pinned production digests in Git on `main`. The cluster also already runs Argo CD. What is missing is the GitOps bridge that turns repository state into cluster state for the `threadr` namespace.

The repository currently contains a single standalone Argo CD `Application` manifest for the control plane, but it is not applied and it does not cover the full namespace-scoped Threadr stack.

## Goals
- Make Argo CD the reconciler for the supported Threadr components in the `threadr` namespace.
- Ensure changes to production digest pins in Git automatically flow into the cluster.
- Keep component boundaries explicit so operators can see sync health per Threadr subsystem.
- Make the GitOps topology idempotent and safe to reapply.

## Non-Goals
- Managing unrelated namespaces or non-Threadr add-ons through this ApplicationSet.
- Replacing the Threadr bot operator's runtime reconciliation of tenant-specific IRC or Discord bot deployments.
- Introducing Argo CD Image Updater in this change.

## Decisions
- Decision: Use an `ApplicationSet` rather than one large `Application`.
  Reason: Threadr already has separate Kustomize roots for CNPG, NATS, control-plane, and worker concerns, and separate applications make sync health and ordering easier to reason about.

- Decision: Scope the first pass to namespace-scoped Threadr components.
  Reason: The user asked specifically for the `threadr` namespace, and the bot operator itself currently lives outside that namespace.

- Decision: Continue using repository-pinned image digests as the deployment source of truth.
  Reason: The existing GitHub Actions workflow already updates the production image patch files on `main`; Argo CD should consume that state rather than introducing a second image mutation mechanism.

## Proposed Topology
The ApplicationSet should generate Applications for:
- `threadr-cnpg` -> `k8s/threadr/cnpg`
- `threadr-nats` -> `k8s/threadr/nats`
- `threadr-control-plane` -> `k8s/threadr/overlays/control-plane/production`
- `threadr-worker` -> `k8s/threadr/overlays/worker/production`

Common expectations:
- destination namespace: `threadr`
- destination server: in-cluster Kubernetes API
- automated sync enabled with prune and self-heal
- `CreateNamespace=true`

Ordering expectations:
- namespace and dependency components reconcile before workloads that depend on them
- control-plane and worker reconcile after the namespace-scoped data-plane dependencies are available

## Risks / Trade-offs
- A misconfigured ApplicationSet could prune live resources unexpectedly.
  Mitigation: keep sources explicit, scope paths narrowly, and rely on Kustomize roots already used by operators.

- If GitHub Actions does not update the production digest pins, Argo CD will reconcile stale images.
  Mitigation: document that the current image-publish workflow is the expected producer of digest changes.

- The bot workloads remain operator-managed and are not directly generated by Argo CD.
  Mitigation: document this boundary clearly so operators do not expect the ApplicationSet to own tenant bot Deployments.

## Migration Plan
1. Add the ApplicationSet manifest and any supporting project or folder structure.
2. Point each generated Application at the existing Kustomize root for that component.
3. Apply the ApplicationSet in the `argocd` namespace.
4. Verify Argo CD creates the expected Applications and reconciles the `threadr` namespace.

## Open Questions
- Should the first pass also manage the bot operator from Argo CD, even though it is not in the `threadr` namespace?
- Should the ApplicationSet use the default Argo CD project, or should Threadr get a dedicated `AppProject` with narrower repository and destination rules?
